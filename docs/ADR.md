# Architecture Decision Records (ADR)

각 결정은 **맥락 → 결정 → 대안 → 결과** 순으로 기록한다. 상태: `Accepted` | `Proposed` | `Superseded`.
새 결정이 기존 결정을 뒤집으면 기존 ADR을 `Superseded`로 표시하고 새 ADR을 추가한다.
배경 설계는 [`ARCHITECTURE.md`](./ARCHITECTURE.md), 제품 기획은 [`PLAN.md`](./PLAN.md) 참고.

## ADR-0001 — 멀티-케이던스 데이터 수집 (빠른 레인 + 느린 레인)
**상태**: Accepted
**맥락**: 13F만 쓰면 분기 단위 + 최대 45일 지연이라 "거물이 지금 뭘 사는지"를 너무 늦게 본다.
**결정**: 13F(느린 레인, 전체 포트폴리오)에 더해 **Form 4(T+2)·13D(T+5)·DART 일반 5%(T+5)** 사건 공시를
빠른 레인으로 함께 수집한다. **13G(기관 패시브)·NPS 국내(약식·특례)는 분기/월 단위라 느린 레인.** 피드 = 두 레인 합집합(`filing_date` 최신순).
**대안**: ① 13F만(단순하지만 느림) ② 유료 실시간 데이터(비용·라이선스).
**결과**: *일부* 핵심 움직임(US 내부자/액티비스트·한국 일반 5%) 지연 45일→2~5일. 빠른 레인은 부분적(내부자·5%↑만), 5%미만·비내부자는 13F뿐. **NPS·13G는 느림** → UI에 케이던스 명시(ADR-0015).

## ADR-0002 — `activity_events` 통합 이벤트 테이블
**상태**: Accepted
**맥락**: Form 4(거래)·13D/G(지분 스냅샷)·DART(지분 변동)는 형태가 제각각이나 피드에선 한 줄로 보여야 한다.
**결정**: 빠른 레인 공시를 공통 `event_type(BUY/SELL/STAKE_*)` 스키마의 `activity_events`로 정규화. 13F 스냅샷은 별도 `holdings`.
**대안**: 소스별 테이블 분리(피드 쿼리가 복잡), holdings에 혼합(의미 충돌).
**결과**: 피드 쿼리가 단순. 소스별 뉘앙스는 nullable 컬럼(`price_per_share`, `pct_of_company_after`, `intent`)으로 보존.

## ADR-0003 — Node.js + TypeScript + Fastify 백엔드
**상태**: Accepted
**맥락**: 사용자가 TS에 익숙(옆 Expo 앱). 인제스트는 주로 HTTP + XML 파싱.
**결정**: 백엔드를 Node/TS + Fastify(스키마-퍼스트). 프론트와 언어·타입 공유.
**대안**: Python(데이터 친화적이나 언어 분리), NestJS(더 무거움).
**결과**: API↔프론트 타입 일원화. 한국 가격(pykrx)만 Python 사이드카로 격리(ADR-0008).

## ADR-0004 — PostgreSQL + Drizzle ORM
**상태**: Accepted
**맥락**: 보유 데이터는 분기 시계열·관계형. 합의/diff는 window 함수(`LAG`) 기반 집합 연산.
**결정**: Postgres + Drizzle(TS-퍼스트, SQL 투명). 가격 볼륨 커지면 TimescaleDB 확장.
**대안**: NoSQL(집합 연산 불리), Prisma(추상화가 두꺼움).
**결과**: 파생 로직을 순수 SQL로 결정적 검증 가능.

## ADR-0005 — BullMQ(Redis) + node-cron 인제스트
**상태**: Accepted
**맥락**: 빠른 레인은 분 단위 폴링, 13F는 분기 집중 + SEC 10 req/s 엄격 제약.
**결정**: BullMQ로 레이트리미터·재시도·동시성 제어, node-cron으로 트리거. 전역 throttle ≤8 req/s.
**대안**: 단순 cron(레이트리미트/재시도 직접 구현 부담).
**결과**: 폴링·백오프·멱등 재시도가 안전. 초기 인프라 무게는 감수(사용자가 풀 아키텍처 선택).

## ADR-0006 — pnpm 모노레포 + `packages/shared` 단일 진실원
**상태**: Accepted
**맥락**: 웹·(추후)모바일·API가 같은 데이터 계약을 공유해야 한다.
**결정**: pnpm workspaces. `packages/shared`에 Zod 스키마 + 추론 타입 + API 클라이언트.
**대안**: 분리 레포(동기화 비용), 타입 중복(드리프트).
**결과**: 계약 위반이 컴파일 타임에 차단. 모바일 추가 시 백엔드 변경 0.

## ADR-0007 — 첫 프론트엔드는 웹(Next.js)
**상태**: Accepted
**맥락**: MVP는 데이터 밀집(피드·랭킹·차트). 반복 개발·공유·검증 속도 중요.
**결정**: 웹(Next.js)부터. API는 플랫폼 무관이라 모바일은 Phase 3에 동일 API 재사용.
**대안**: Expo 모바일 우선('앱' 느낌이나 데이터/차트 반복 개발 느림).
**결과**: SEO·공유 링크 확보, 빠른 검증. 모바일은 후속.

## ADR-0008 — 미국 먼저, 한국 fast-follow
**상태**: Accepted
**맥락**: 13F가 데이터가 가장 정제·풍부. DART는 회사중심이라 투자자중심 재구성이 까다로움.
**결정**: MVP는 미국 완성도 우선, 한국(DART)은 같은 스키마/피드로 즉시 fast-follow.
**대안**: 미국·한국 동시(완성도↑이나 첫 버전 지연).
**결과**: 빠른 첫 버전. 스키마·API는 처음부터 시장 무관.

## ADR-0009 — CUSIP→티커 내부 매핑 전략
**상태**: Accepted
**맥락**: 13F는 CUSIP만 제공, 무료 완전 CUSIP↔티커 마스터는 없음(CGS 라이선스).
**결정**: 내부 `cusip_map`을 (오픈데이터 시드 + `nameOfIssuer` 퍼지매칭 + 수동 큐레이션)으로 구축, `confidence` 추적.
**대안**: 유료 CUSIP 라이선스(비용), 매핑 포기(티커 없이 UX 빈약).
**결과**: 큐레이션 10~20명 보유 한정의 **유한 문제**라 감당 가능. CUSIP 목록 자체는 게시 금지.

## ADR-0010 — NPS를 한 투자자 + 두 소스 피드로 모델링
**상태**: Accepted
**맥락**: 국민연금은 미국분(EDGAR 13F)·국내분(DART)을 따로 공시 → 합의 집계에서 이중 카운트 위험.
**결정**: `parent_investor_id` 자기참조로 두 소스 행을 한 투자자로 묶어 UI는 통합 프로필, 합의는 단일 집계.
**대안**: 완전 별개 투자자(이중집계), 단일 행(소스 구분 불가).
**결과**: 정확한 합의 + 통합 프로필.

## ADR-0011 — 성과/성공률 = filing_date 기준 초과수익
**상태**: Accepted (구현 Phase 2)
**맥락**: "성공률"을 정직하고 방어 가능하게 정의해야 한다. quarter-end는 45일간 알 수 없음.
**결정**: **filing_date**(공개 후 행동 가능 최초 시점) 기준 forward return − benchmark(S&P500/KOSPI) = 초과수익. 승률 = 초과수익>0 비율.
**대안**: quarter-end 기준(비현실적), 절대수익(시장 효과 미보정).
**결과**: "공개 롱북을 따라갔다면 지수를 이겼는가"의 honest 지표. 실현손익 아님을 UI 명시. 스키마는 지금 대비. **데이터는 무료·지연(EOD)로 제한**(상업 라이선스 비용 회피). **주의: 생존자 편향·벤치마크 불일치·45일 지연이 알파 잠식 → '예측' 아닌 '정보', 주장 전 백테스트**(ADR-0015).

## ADR-0012 — 에이전틱/하네스 엔지니어링 개발 방법론
**상태**: Accepted
**맥락**: 이 프로젝트는 AI 에이전트(Claude Code)가 주도해 구현·유지한다. 에이전트가 안전·빠르게 일하려면 결정적 검증이 필요.
**결정**: ① 작은 수직 슬라이스 ② 파서/파생은 순수 함수·순수 SQL ③ 실제 공시 **fixture 골든 테스트** 필수 ④ 멱등 인제스트 ⑤ `packages/shared` 계약 우선 ⑥ `typecheck && lint && test` + 레이트/멱등 통과가 "완료". (Phase 2) 추천 로직 eval 골든셋.
**대안**: 수동 검증/통합테스트만(에이전트 자가검증 어려움, 회귀 취약).
**결과**: 에이전트가 자가검증·재시도 가능. 회귀 방지. 운영 지침은 루트 `CLAUDE.md`.

## ADR-0013 — 투자 자문 면책 & 중립 카피
**상태**: Accepted
**맥락**: 큰손 매매를 "추천"으로 보여주면 투자 자문으로 오해·법적 노출 위험.
**결정**: 모든 화면·API 응답에 **면책 문구** 상시. 카피는 중립("공개된 매수", "합의 보유")만, "사세요"류 금지. `disclaimer` 필드 표준화.
**대안**: 면책 생략(법적 위험), 자문성 표현(규제 위험).
**결과**: 정보 제공 서비스로 포지셔닝. UX 카피 가이드가 됨. **추가(감사 반영)**: 제품을 **'추천' 아닌 '공시 정보 집계'**로 리프레이밍하고 **철저히 비개인화(불특정 다수·동일 콘텐츠)** 유지. 한국 **유사투자자문업**(유료 추천=신고 의무, 개인화=투자자문업 등록·미등록 형사처벌 **3년/1억**)·미국 **발행인 면제(Lowe v. SEC)** 경계 준수. 유료화·개인화 도입 전 **법률 자문**.

## ADR-0014 — 테스트 우선(TDD) 정책 + 기계적 백스톱
**상태**: Accepted
**맥락**: "무조건 테스트 먼저"를 원하지만, git diff는 최종 상태만 보여 **시간상 순서는 증명 불가**(테스트·구현을 한 커밋에 넣으면 순서 불명).
**결정**: **2층 강제**.
① **에이전트 프로토콜**(`CLAUDE.md`): RED→GREEN→REFACTOR, 실패 테스트 우선·실행결과 제시. 이 프로젝트는 에이전트가 코드를 작성하므로 *진짜 test-first가 여기서* 일어남.
② **기계적 백스톱**(`.githooks/pre-push`): 푸시되는 커밋 범위에서 **소스(`apps|packages|services` 코드) 변경 시 테스트 동반을 강제**(소스만 변경 + 테스트 없음 → 차단). 예외는 `[skip-tdd]`/`SKIP_TDD=1`.
**대안**: 커버리지·패치 커버리지 게이트(우회 어렵지만 Phase 0 스캐폴딩 필요 → **후속 보류**), 순서 증명 시도(불가능), 프로토콜만(백스톱 없음 → 회귀 취약).
**결과**: 순서는 못 박아도 **"테스트 없는 코드 머지"는 기계적으로 차단**. 클라이언트 훅은 `--no-verify`로 우회 가능 → 권위 보장은 추후 **CI 패치 커버리지 · GitHub 브랜치 보호**로 보완(현재 미적용).

## ADR-0015 — 감사 후 정정: 케이던스 현실·식별자·통화·데이터 라이선스·성과 범위
**상태**: Accepted
**맥락**: 1차 출처(sec.gov·opendart·법령) 감사에서 여러 전제가 부정확하거나 비용·법적 리스크를 동반함이 확인됨.
**결정**:
- **케이던스**: 빠른 레인 = US Form 4(T+2)·US 13D(T+5)·KR 일반 5%(T+5). **NPS 국내(약식·특례)·13G(기관 패시브)는 느린 레인**(분기/월). `dataAsOf`로 케이던스 명시, "NPS 실시간" 류 카피 금지.
- **파싱**: Form 4 **Table II(파생/워런트) 포함**, 13F **상장옵션(putCall) 포함**, `corpCode.xml`은 **ZIP**(압축해제), `elestock`은 `sp_stock_lmp_*` 스키마, 다중 보고인 이중집계 주의.
- **식별자**: CUSIP 해결에 **OpenFIGI(무료)** 추가, 기업이벤트로 식별자 **시점 버전** 관리.
- **통화**: `fx_rates` 추가 — KR `value_usd` = 가격×수량×FX.
- **데이터 라이선스/비용**: pykrx·FMP/Finnhub 무료 티어는 **상업 재배포/표시 불가** → 성과 기능은 **무료·지연 데이터로 제한**(필요 시 KRX/유료 라이선스 예산).
- **포지셔닝/법률**: **미국 먼저 유지**(레드오션 인지). '추천'→'정보 집계' 리프레이밍 + 비개인화(ADR-0013).
- **CI**: `build`는 pre-push에서 제거 → CI.
**대안**: 기존 전제 유지(부정확·법적/비용 리스크 방치).
**결과**: 기획이 1차 출처와 일치. 전략 결정(포지셔닝·성과 범위·법률 프레이밍)이 문서에 반영됨.

## ADR-0016 — 테스트 레이어 정책: 단위/픽스처 우선, 화면 불변식만 E2E(Playwright)
**상태**: Accepted
**맥락**: 화면(웹) 검증을 위해 Playwright(MCP + `@playwright/test`)를 도입했다. 그러나 프로젝트의 가치·위험은 대부분 백엔드 파이프라인(파서·인제스트·파생)에 있고, 거긴 빠르고 결정적인 vitest 픽스처가 정답이다. 모든 테스트를 E2E로 돌리면 느리고(브라우저+서버 기동) 깨지기 쉬워 TDD 루프만 둔화된다.
**결정**: **테스트 피라미드 3층**으로 분리한다.
① **단위/픽스처(vitest)** — 기본·최다. 파서(Form4·13D/G·13F·DART)·파생(`compute:changes`·`compute:consensus`)·멱등성·레이트리밋. (ADR-0012)
② **컴포넌트(vitest + jsdom)** — 중간. 브라우저 불필요한 UI 로직(포맷·empty/loading·면책 컴포넌트 렌더). *Phase 1에 jsdom 프로젝트 추가 예정.*
③ **E2E(Playwright)** — 가장 적게. **브라우저로만 잡히는 화면 불변식만**: 면책 문구 상시 노출(ADR-0013)·중립 카피 가드레일·차트 렌더(Recharts 하이드레이션)·핵심 페이지 풀스택 스모크.
**트리거 룰**: 데이터·파싱·파생 슬라이스 → vitest(기존과 동일). 렌더 동작·가드레일·차트·핵심 플로우 → 해당 E2E 스펙 추가/확장. E2E 스펙은 루트 `e2e/`에만 두어 vitest(`{apps,packages}/*/{test,src}/**`)와 글롭을 분리한다.
**강제 지점**: E2E는 무거워 `.githooks/pre-push`에서 **기본 비활성(opt-in `RUN_E2E=1`)**, **상시 강제는 CI**(`build`를 CI로 옮긴 ADR-0015와 같은 논리). 개발 중 시각 검증은 Playwright **MCP**(`.mcp.json`)로, 검증된 시나리오는 E2E 스펙으로 박제.
**대안**: 전면 E2E(느리고 flaky, 루프 둔화), E2E 없음(면책 등 화면 가드레일 회귀를 못 잡음), 수동 눈검증만(회귀 취약·재현 불가).
**결과**: 빠른 단위 피드백을 유지하면서 **화면 전용 가드레일(특히 ADR-0013 면책)을 결정적으로 고정**. 첫 E2E 스펙은 `e2e/disclaimer.spec.ts`(면책 푸터). 룰은 `CLAUDE.md` 하네스 섹션에 반영.

## ADR-0017 — SEC fetch 레이어의 결정적 레이트리미터(bottleneck 대신 주입 Clock)
**상태**: Accepted
**맥락**: 가드레일은 **SEC ≤8 req/s + User-Agent 필수 + 429 0건**을 요구하고, 하네스는 모든 변경의 **결정적 검증**을 요구한다(ADR-0012). PLAN 의 기술 선택은 `bottleneck`(전역 ≤8 req/s)이었으나, bottleneck 은 실제 타이머·내부 상태로 도는 라이브러리라 간격·백오프를 **결정적으로 단위테스트하기 어렵다**(실시간 대기·flaky). ADR-0005 의 BullMQ 는 *잡* 단위 재시도·동시성을 다루지, *per-request* 레이트 프리미티브가 아니다.
**결정**: SEC fetch 레이어를 **주입 가능한 `Clock`**(now/sleep) 위에서 도는 자체 프리미티브로 구현한다.
- `RateLimiter`: 단일 프로미스 체인으로 작업을 줄세워 **디스패치 간격 ≥ minIntervalMs**(=125ms → 8 req/s) 보장. 간격은 슬롯 확보 시점 기준이라 작업 지속시간과 분리.
- `EdgarClient`: 모든 요청을 limiter 로 통과 + **UA 미설정 시 생성자 fail-loud** + 요청별 **타임아웃 AbortSignal** + 429/5xx·네트워크 오류 **백오프 재시도**(`Retry-After` 우선·상한 클램프, 없으면 cap·지터 지수 백오프), 비재시도 4xx 즉시 throw, **HTTP 상태 소진 시 `EdgarHttpError`·네트워크 오류 소진 시 원오류 전파**, 폐기 응답 body cancel. `fetch`·`clock`·`limiter`·`random` 주입으로 결정적 테스트.
- **전역성은 기본값으로 강제(safe-by-default)**: `clock`/`limiter`/`minIntervalMs` 미주입 시 **모듈 공유 `sharedLimiter`** 를 쓴다 → fast/slow 레인 폴러가 각자 `createEdgarClient()` 를 불러도 합산 ≤8 req/s. 호출자 기억력에 의존하지 않음(주입 시에만 분리 — 테스트 격리용).
- 역할 분담: **BullMQ(ADR-0005)=잡 큐·재시도·동시성**, **이 프리미티브=per-request 레이트·UA·HTTP 백오프**. 둘은 보완.
**대안**: ① bottleneck 채택(실시간 의존 → flaky·결정적 테스트 곤란, 하네스 가치와 충돌). ② BullMQ 레이트리미터에만 의존(잡 레벨이라 단일 요청 간격·UA·HTTP 상태 처리를 못 덮음, 단위테스트 불가). ③ 가짜 타이머(`vi.useFakeTimers`)로 bottleneck 테스트(설정 복잡·내부 구현 의존, 깨지기 쉬움).
**결과**: ≤8 req/s(전역 공유 limiter 포함)·UA·429 백오프·Retry-After 클램프·타임아웃이 **가짜 시계 주입으로 0ms·결정적**으로 검증됨(`rate-limiter.test.ts`·`edgar-client.test.ts`). `bottleneck` **도입 회피**(애초에 package.json/lock 에 추가하지 않음 — 실시간 의존 라이브러리 대신 자체 프리미티브). fast/slow 레인 폴러는 이 클라이언트를 공유 진입점으로 사용.
